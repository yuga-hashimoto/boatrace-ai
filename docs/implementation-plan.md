# boatrace-ai 実装計画

この文書は、提示されたエグゼクティブサマリーを現行リポジトリの実装状況に合わせて再構成したものです。
目的は「将来の全体像」と「いまコードに入っているもの」を混同しないことです。

## 1. 目的

- ボートレースの過去データ、当日出走表、直前情報、オッズ、外部環境データを統合し、着順確率と投票期待値を予測する。
- 予測モデルだけでなく、収集、前処理、評価、バックテスト、配信、運用までを一貫したシステムとして整備する。
- 最終KPIは単なる精度ではなく、時系列検証を通過した期待値ベースの回収率と運用品質に置く。

## 2. 現在の実装範囲

| 領域 | 現在の状態 |
| --- | --- |
| 公式データ収集 | 公式HTMLから `race index` `racelist` `beforeinfo` `raceresult` `odds3t` を収集済み |
| 公式ダウンロード | 日次 `B` / `K` LZH を取得して raw JSON / SQLite に展開済み |
| 履歴保存 | 1レース1JSONで `data/raw/YYYYMMDD/` に保存済み |
| 履歴DB | SQLite に `1レース1行` で保存し、過去1年分を蓄積しやすい形に移行済み |
| データセット生成 | entrant単位CSVを `data/processed/entrants.csv` に生成済み |
| 学習 | `HistGradientBoostingClassifier` による勝率モデルを実装済み |
| 推奨ロジック | 3連単候補を確率とオッズから評価し、ROI基準で推薦済み |
| 評価 | holdout と walk-forward のROI指標を実装済み |
| 資金管理評価 | flat / Kelly の bankroll シミュレーションを backtest CLI で実装済み |
| 配信 | CLI と予測JSON、venue全12R予測、note向けHTML生成まで実装済み |
| テスト | parser、dataset、train、history、note の自動テストあり |

## 3. 計画文書からの修正点

以下は、そのまま実装済みとして扱うと誤解になるため、方針として取り込む項目です。

- 公式ダウンロードは一部実装です。
  現在は日次 `B` / `K` の取り込みまで実装済みで、期別成績など他アセットの自動投入は未実装です。
- REST API、GraphQL、Web UI、S3、Airflow、監視基盤、CI/CD は未実装です。
  これらは「目標アーキテクチャ」であり、現状成果物ではありません。
- 3連単確率は直接学習していません。
  現在は艇ごとの勝率からPlackett-Luce型で3連単候補を導出しています。
- オッズ履歴は完全アーカイブではありません。
  収集済みレースの `odds3t` 保存と補完はありますが、長期フルヒストリは今後の収集対象です。
- Teleboat 等の会員制・私的APIは既定路線にしません。
  導入する場合は別途、利用条件と法務レビューが必要です。
- 法務・利用規約は「実装前提」ではなく「運用前提」です。
  商用利用や自動化の可否は、実装より前に確認が必要です。

## 4. 目標アーキテクチャ

### 4.1 データレイヤ

- `data/raw/`
  公式HTMLや外部ソースの生データを保存する一次保管領域。
- `data/external/history.sqlite`
  過去1年分以上の履歴を query しやすく保つ SQLite ストア。
- `data/processed/`
  学習・評価に使うテーブル化済みデータ。
- 将来拡張:
  RDB、オブジェクトストレージ、特徴量ストアを追加し、再学習と監査を容易にする。

### 4.2 学習・評価レイヤ

- 勝率モデルを基盤に、3連単候補生成、EV算出、回収率評価を行う。
- 検証は時系列 holdout と walk-forward を両方使う。
- モデル学習は全ての完全な venue-day を使う。
- betting policy 選定と walk-forward 安定性チェックは、直近連続開催ブロックを優先する。

### 4.3 配信・運用レイヤ

- 現状:
  CLI と JSON/HTML 出力。
- 将来:
  REST API、Web UI、ジョブスケジューラ、監視、モデルレジストリを追加する。

## 5. データソース方針

| ソース | データ | 用途 | 優先度 | 現在の状態 |
| --- | --- | --- | --- | --- |
| BOAT RACE公式HTML | 出走表、直前情報、結果、3連単オッズ | 当日予測、履歴収集 | 最優先 | 実装済み |
| BOAT RACE公式ダウンロード | 日次番組表 `B`、日次成績 `K`、期別成績など | 長期履歴の一括投入 | 高 | `B` / `K` は実装済み、他は未実装 |
| 各場モーター情報 | モーター履歴、交換情報 | 特徴量強化 | 高 | 未実装 |
| 気象庁データ | 風、気温、気圧、降水、潮位 | 環境特徴量 | 中 | 未実装 |
| 払戻・オッズ履歴 | 市場情報、期待値検証 | ROI最適化 | 最優先 | 部分実装 |
| 外部/有料API | 補完データ | 収集安定化 | 低 | 未採用 |

## 6. 特徴量ロードマップ

### 6.1 実装済み

- 艇番
- 級別
- 年齢、体重、F/L
- 全国/当地勝率、2連率、3連率
- モーター/ボート連対率
- 平均ST
- 展示タイム、チルト、調整体重、展示ST
- 気温、風速、水温、波高、風向
- 部品交換数、プロペラ注記

### 6.2 次に追加するもの

- 選手の直近N走集計
- 当地・同開催節間の成績集計
- 進入想定と進入変化
- 枠番と場特性の交互作用
- オッズ分布と人気偏差
- 潮位・潮流・時間帯
- モーター使用履歴の時系列要約

### 6.3 後段で検討するもの

- レーティング指標
- SHAPで残った相互作用特徴
- calibration用の補助特徴
- 3連単直接ランキング用特徴

## 7. モデル方針

### 7.1 現在

- 主モデル:
  `HistGradientBoostingClassifier`
- 出力:
  各艇の勝率
- 3連単:
  勝率から順序確率を導出

### 7.2 拡張候補

- LightGBM / XGBoost
- CatBoost
- 1着分類 + 2着/3着条件付きモデル
- Listwise / pairwise ranking
- calibration レイヤ
- アンサンブル

### 7.3 評価指標

- `log_loss`
- `brier_score`
- `roc_auc`
- `top1_hit_rate`
- `winner_in_top3_rate`
- `recommendation_roi`
- `walk_forward_recommendation_roi`
- `daily_roi_floor`
- `daily_roi_median`

## 8. ベッティング戦略方針

- 精度指標だけでなく、推薦ロジック込みの回収率を主指標にする。
- 購入点数は広げすぎず、まずは `1レース1点` を基準にする。
- EVだけでなく `min_probability` と `min_edge` を使って薄い買い目を抑える。
- すでに以下は実装済み。
  - fractional Kelly
  - capped Kelly
  - 日次 stop-loss / take-profit
  - レース単位の exposure 制限
- 将来は以下を段階導入する。
  - 会場別上限
  - 資金曲線ベースのドローダウン制御
  - 税・控除率前提を含む実運用バックテスト

## 11. KPI

- 直近 holdout の `recommendation_roi >= 1.2`
- recent walk-forward の `recommendation_roi >= 1.2`
- 不完全 venue-day を学習に混ぜない
- 収集失敗率を継続的に下げる
- 予測生成を開催日内に安定完了させる

## 12. 法務・運用メモ

- 公式サイト・各場サイト・外部APIの利用条件は、本番運用前に確認する。
- 商用利用や再配布の可否は実装判断ではなく運用判断として管理する。
- 予測結果は参考情報であり、結果保証ではないことを明示する。
- 自動投票や過度な射幸性を前提にした機能は既定では採用しない。
